Техническое задание v1.2 — «Field Service» (две Telegram‑бота: мастер-бот и админ-бот)
Дата: 2025‑09‑15
Статус: v1.2 (frozen)
Цель: Внутренняя система обработки заявок на выезд мастеров. Заявки создаются и ведутся через админ-бот; мастера работают через мастер-бот.
1. Роли и доступ
•	Мастер: Регистрация (онбординг) через бот с согласиями и документами; возможность включать/выключать рабочую смену и брать перерыв; получение офферов (предложений заявок) в соответствующих районах; прохождение стадий заказа («В пути», «Работаю»); закрытие заказа (ввод суммы и загрузка акта выполненных работ); оплата комиссии за заказ.
•	Логист: Работа с очередью заявок – мониторинг новых заявок, назначение мастеров (автоматически или вручную) в пределах закрепленных городов.
•	Админ (городской): Полномочия логиста по своим городам, плюс модерация анкет мастеров, управление финансами/комиссиями и получение отчетов по своим городам.
•	Админ (глобальный): Полный доступ без ограничений по городам; управление всем персоналом/ролями и глобальными настройками (напр., реквизиты для оплаты).
Ограничение видимости данных: Администраторы и логисты городского уровня видят данные только по своим городам; глобальный админ видит все. Мастер видит только свои личные данные и заказы.
Доступ по кодам: Глобальный администратор выдает специальные коды для привязки ролей персонала: CITY_ADMIN, LOGIST, GLOBAL_ADMIN. При генерации кода для городских ролей указывается город (или несколько) закрепления. Используя код, сотрудник авторизуется в админ-боте и получает соответствующие права.
2. География
2.1. Города (фиксированный список)
Система поддерживает фиксированный перечень городов (79 позиций: Москва, Санкт‑Петербург, Новосибирск, …, Шахты). Выбор города осуществляется только из этого списка – ввод произвольного города, не включенного в список, не допускается.
2.2. Районы и улицы
•	Районы: Для выбранного города используется справочник административных районов. Если районов больше 5, то список выводится с пагинацией (по 5 районов на страницу кнопок). Если справочник районов для города отсутствует, оператор может сразу перейти к вводу улицы или запросить геопозицию.
•	Улицы: При указании улицы в адресе используется поиск с помощью библиотеки RapidFuzz (нечеткий поиск по справочнику улиц города). Предлагаются наиболее похожие варианты; также доступен вариант ручного ввода улицы (от 2 до 50 символов) или выбор «Нет улицы» (для адресов без названия улицы, например коттеджные поселки).
2.3. Координаты без внешних API
Внешние геокодеры (например, Яндекс) не используются. Координаты (широта/долгота) адреса определяются внутренне: - Если при создании заявки администратор отправил геолокацию – используются ее координаты. - Иначе система берет координаты центроида улицы (если улица найдена в справочнике); если улицы нет – берется центроид района; если нет района – центроид города.
- Координаты нужны для отображения адреса в интерфейсе и отчётах. В алгоритме распределения координаты не применяются (см. §7), т.е. удаленность мастера не учитывается при выборе кандидатов.
(Примечание: Координаты адресных объектов хранятся в локальной базе. При импорте справочников городов/районов/улиц используются центроиды; дубликаты исключены, пороги похожести для RapidFuzz соответствуют методике исходного проекта.)
3. Время и слоты
•	Рабочие часы сервиса: Ежедневно с 10:00 до 20:00 по локальному времени города заявки. В этот интервал могут назначаться визиты мастеров.
•	Отложенные заявки: Заявки, созданные после 20:00, получают статус DEFERRED (отложена) и автоматически переводятся в поиск мастера в 10:00 следующего дня (по времени соответствующего города).
•	Слоты визита: При создании заявки указывается желаемое время визита. Доступны варианты: «ASAP» (как можно скорее), «Сегодня» или «Завтра», с выбором окна: 10–13, 13–16, 16–19.
•	Late ASAP: Если заявка помечена как ASAP после 19:30, админ-бот запрашивает подтверждение переноса на следующий день утренний слот (10–13 следующего дня), так как мастера уже могут не успеть в тот же день.
4. Статусы заказа
Последовательность основных статусов заявки (order.status):
CREATED → SEARCHING → ASSIGNED → EN_ROUTE → WORKING → PAYMENT → CLOSED.
Дополнительные особые статусы:
- GUARANTEE (гарантийная заявка – по завершённому заказу, см. §10),
- DEFERRED (отложена до начала рабочего дня),
- CANCELED (отменена).
Каждое изменение статуса фиксируется в истории (таблица order_status_history с полями: кто перевел, когда, предыдущий статус, новый статус, причина/комментарий если есть). Это обеспечивает прослеживаемость и аудит прохождения заказа.
5. Категории работ
Поддерживаются следующие категории работ (типовые направления задач мастеров): - ELECTRICS (электрика)
- PLUMBING (сантехника)
- APPLIANCES (бытовая техника)
- WINDOWS (окна и стеклопакеты)
- HANDYMAN (мелкий бытовой ремонт)
- ROADSIDE (автопомощь на дороге)
Категория заказа указывается при создании и используется для подбора мастера с соответствующими навыками.
6. Создание заявки (через админ-бот)
Создание новой заявки происходит через интерфейс админ-бота (только для логиста/админа). Процесс включает несколько этапов:
1.	Контакт клиента: Ввод имени, фамилии клиента и телефона. Телефон вводится в форматах +7XXXXXXXXXX или 8XXXXXXXXXX – при сохранении он нормализуется в формат +7.... Если формат некорректен, выдается ошибка и запрос повторяется (пример ошибки: «Неверный формат телефона. Пример: +7XXXXXXXXXX или 8XXXXXXXXXX»).
2.	Адрес: Выбор адреса проходит поэтапно: сначала город (из списка поддерживаемых, с возможностью поиска по названию), затем подтверждение выбора. Далее бот предлагает выбрать район (списком с пагинацией, если районов >5) и улицу: либо выбрав из предложенных по поиску (RapidFuzz), либо введя вручную, либо отметив «Нет улицы». После выбора или ввода улицы, бот запрашивает номер дома (строка 1–10 символов: цифры и буквы, допускаются символы «/», «к» – корпус, «с» – строение). В конце адресного ввода отображается полный адрес для подтверждения пользователем. Если район не был выбран (например, город без районов или пропущен), заявка помечается флагом no_district (см. §7.2).
3.	Описание проблемы: Текстовое описание заявки от клиента (от 10 до 500 символов). Админ-бот контролирует длину текста. Дополнительно можно прикрепить вложения – фото и/или документы (например, снимки поломки, техническое задание и т.д.). Вложения необязательны; бот принимает несколько фото или файлов по желанию. (Примечание: Внутри система хранит файлы, переданные через Telegram, по их file_id – уникальному идентификатору файла на серверах Telegram, что позволяет впоследствии пересылать или скачивать файл без повторной загрузки[1].)
4.	Выбор слота: Администратор указывает желаемое время визита мастера. Как отмечено в §3, доступны опции ASAP, сегодня или завтра с выбором окна (10–13, 13–16, 16–19). Бот проверяет правила: если текущее время за пределами рабочего или заявка ASAP поступает слишком поздно, предложит перенос.
5.	Создание заявки: После подтверждения всех данных админ завершаeт ввод. На этом этапе заявка записывается в базу данных со статусом CREATED, а затем автоматически переходит в статус SEARCHING, запускается процесс поиска мастера (исключение – если заявка создана после 20:00, тогда статус сразу DEFERRED до 10:00 следующего дня, см. §3).
6.	Карточка заказа: Админ-бот показывает резюме созданной заявки – полный адрес, категория, описание, приложенные файлы, выбранный слот, а также контакты клиента. Далее эта заявка появляется в очереди логиста/админа. Дальнейшая работа с ней идет по процессам распределения (§7), назначения (§13) и выполнения мастером (§8).
Важно: Если у заявки отсутствует выбранный район (флаг no_district), автоматическое распределение не выполняется – такая заявка сразу направляется в очередь логиста для ручного назначения мастера.
7. Автоматическое распределение заявок
Автораспределение отвечает за автоматическую отправку офферов мастерам, подходящим под новую заявку, без участия человека (логиста) в первые минуты после создания заявки.
7.1. Циклы распределения: тики, SLA, раунды, эскалации
•	Тик (tick): Периодический цикл проверки новых заявок. Фоновый планировщик каждые 30 секунд отбирает все заявки в статусе SEARCHING (в активном поиске мастера) и пытается назначить их. (Планировщик реализуется внутри сервиса: например, ассинхронная задача, регулярно просматривающая базу).
•	Оффер и SLA: В каждом цикле для каждой заявки формируется список кандидатов (см. фильтры §7.2 и сортировку §7.3). Системa предлагает заявку одному мастеру – отправляет ему оффер через мастер-бот (карточка заявки с кнопками «Взять заказ» или «Отказаться»). После отправки оффера запускается таймер SLA (Service Level Agreement) – в течение 120 секунд мастер может ответить. На этот период заявка ожидает ответа и другим мастерам не предлагается.
•	Раунды: Автораспределение проводит 2 раунда офферов. Если первый кандидат отказался или не ответил за SLA-время, система переходит ко второму кандидату в следующем цикле (или сразу, если ответ получен раньше таймаута). После двух безуспешных попыток заявка эскалируется на ручное управление.
•	Эскалация: Если автоматическое назначение не удалось (2 раунда офферов отклонены/просрочены), заявка передается логисту – ее статус остается SEARCHING, но авто-офферы прекращаются. Логист получает уведомление или видит в интерфейсе, что заявке требуется ручное вмешательство. Если логист не назначит мастера в течение 10 минут с момента эскалации, система может отправить дополнительный алерт городскому администратору. Таким образом, максимальная задержка на автоматическое распределение – ~2*120 сек SLA + небольшие ожидания, и еще 10 минут на реакцию логиста.
Все события распределения журналируются в лог (см. Приложение A). Каждый тик записывает информацию: идентификатор заказа, город/район, категорию, тип (NORMAL/GUARANTEE), текущий раунд, значение SLA, число подходящих кандидатов, топ кандидатов с их параметрами и рейтингом, кому отправлен оффер и дедлайн ответа, а также пометки о таймауте/отказе или принятии оффера. Особые случаи (нет района, гарантийный заказ, эскалация) также отмечаются.
7.2. Жёсткие фильтры кандидатов (без учета дистанции)
Мастер попадает в список кандидатов на заявку, только если он удовлетворяет всем следующим условиям:
•	is_on_shift = true (мастер находится на смене и не в перерыве). Мастера вне смены или на перерыве не получают авто-офферы. (На перерыве внутренний флаг is_on_shift ставится false, см. §8).
•	Город мастера совпадает с городом заявки.
•	Район заявки входит в список районов обслуживания мастера (мастер указывает районы при регистрации).
•	Статус мастера: verified = true (прошёл модерацию), is_active = true (не деактивирован), is_deleted = false (не удалён). Заблокированные или неактивные мастера не участвуют.
•	Не превышен лимит максимального числа активных заказов у мастера. По умолчанию мастер не может одновременно вести более 5 активных заказов (лимит настраивается глобально). Если у мастера уже есть 5 незакрытых заказов, новые ему не предлагаются.
•	Категория заявки входит в навыки мастера. При регистрации мастер отмечает категории работ, которые он выполняет; заявка вне его компетенций не будет направлена этому мастеру.
Если у заявки не указан район (флаг no_district), автоматическое распределение пропускает ее (кандидаты не ищутся) – заявка сразу уходит на ручное назначение логисту.
Примечание: В данной версии алгоритма географическое расстояние не учитывается. Все мастера в городе и нужном районе равноправны при отборе кандидатов (отсутствуют расчёты удаленности или штрафы за расстояние).
7.3. Ранжирование кандидатов (сортировка)
Кандидаты, прошедшие фильтры, сортируются по простому правилу с приоритетами (в порядке убывания значимости ключа):
1.	Наличие автомобиля (has_car): Мастера с автомобилем получают приоритет выше пеших.
2.	Средний чек за неделю (avg_week_check): Чем выше средний чек мастера за последнюю неделю, тем выше его позиция. Этот показатель может отражать эффективность/опыт мастера.
3.	Средняя оценка (rating_avg): Средний рейтинг мастера от клиентов (по 5-балльной шкале, см. §11). Более высокие оценки повышают позицию.
4.	Случайный шум: Небольшое случайное число от 0 до 0.9 добавляется к сортировочному ключу, чтобы при прочих равных немного перемешивать список и не допускать постоянного фаворита при равных показателях.
Сортировка производится по ключу, составленному на основе вышеуказанных критериев (в реализации можно комбинировать их в едином численном значении или сортировать последовательно). Мастер с наивысшим итоговым значением ключа становится первым кандидатом на оффер.
(Замечание: на текущем этапе влияние геодистанции отсутствует, хотя потенциально можно расширить алгоритм позднее, например, penalize мастеров далеко от адреса. В данной версии расстояние игнорируется для упрощения.)
7.4. Гарантия (приоритет предыдущего мастера)
Если заявка типа GUARANTEE (гарантийная, см. §10), то алгоритм распределения сначала принудительно ставит первым кандидатом того мастера, который выполнял исходный заказ. То есть при гарантийном повторном вызове предпочтение отдается мастеру, ранее выполнявшему работу, чтобы он сам устранил недостатки. Однако мастер также должен пройти все фильтры: быть на смене, в том же городе/районе, не заблокирован и т.д. Если этот мастер отказывается или не отвечает в SLA, то:
- Его анкета блокируется автоматикой (так как он отказался от гарантийного обязательства, это считается серьезным нарушением, см. §10).
- Далее заявка распределяется обычным образом среди остальных мастеров по алгоритму сортировки, как обычная (т.е. без дальнейших привилегий).
7.5. Логи распределения
Каждый тик распределения логируется строкой в специальный канал/файл логов. Формат логов описан в Приложении A. Здесь хранится информация о процессе отбора кандидатов и принятии решения по офферу на каждом этапе. Примеры полей в логе: order, город/район, категория, тип заявки, раунд, SLA, число кандидатов, отсортированный список топ-кандидатов (с их признаками и значением ключа), решение (кому выслан оффер и до какого времени), а также особые пометки (например, no_district – авто-распределение не выполнялось; force_first – гарантийный мастер поставлен первым; skip с указанием причин пропуска мастеров: off_shift, break, active_limit и др.).
8. Мастер: онбординг, смена и работа с заказами
Интерфейс мастер-бота предназначен для мастеров, выполняющих заявки. Основные функции мастер-бота:
•	Онбординг мастера (регистрация): Мастер запускает бота и вводит код доступа (выданный администратором). Далее бот запрашивает согласие на обработку персональных данных (см. Приложение B.1 для текста соглашения) – без согласия регистрация невозможна. Затем мастер последовательно вводит Фамилию, Имя, Отчество (каждое 2–30 символов, только кириллица, пробел или дефис), телефон (формат проверяется как для клиента, нормализуется в +7…), выбирает свой город из списка поддерживаемых, отмечает один или несколько районов, где готов работать, указывает наличие автомобиля (да/нет), отмечает галочками категории-навыки, которые выполняет (см. §5). После этого бот запросит загрузить фото/скан паспорта (фото или PDF), сделать селфи (фото для удостоверения личности) – эти файлы хранятся как file_id в базе. Далее мастер выбирает способ получения выплат комиссии: карта, СБП или наличные. В зависимости от выбора бот запросит реквизиты:
•	При выборе карты – ввод номера карты (или загрузка фото карты) и ФИО держателя, названия банка.
•	При выборе СБП – номер телефона, привязанного к СБП (с маскировкой в базе).
•	При выборе наличных – ничего дополнительно (предполагается расчёт наличными с админом).
(Реквизиты владельца (админа) для оплаты комиссий хранятся отдельно и не путать с реквизитами мастера.)
Опционально мастер может отправить свою домашнюю геолокацию (для информации, не используется в алгоритмах). После заполнения всех данных мастер завершает регистрацию и переводится в статус «На модерации». Его анкета становится доступна админам для проверки.
•	Управление сменой: После одобрения анкеты (админ меняет verified=true), мастер может приступать к работе. В главном меню мастер-бота есть раздел «Смена», где доступны команды: Включить смену, Выключить смену, Перерыв на 2 часа.
•	Когда мастер включает смену, флаг is_on_shift становится true, и он становится видим для авто-распределения (при условии, что не находится на перерыве).
•	Выключение смены ставит is_on_shift=false (мастер не получает новых офферов).
•	Перерыв: Мастер может взять перерыв продолжительностью 2 часа. Перерыв можно начать только если смена сейчас включена. На время перерыва мастер исключается из распределения (is_on_shift=false), а в системе фиксируется время начала перерыва. По истечении 2 часов перерыв автоматически заканчивается и, если мастер не отключил смену, статус смены возвращается к активному (он снова доступен для заявок). Мастер также может самостоятельно выйти из перерыва раньше времени, включив смену вручную.
•	Получение новых заказов (офферы): В меню «Новые заявки» мастер видит список доступных ему офферов (заявок в статусе SEARCHING, отправленных этому мастеру). Список может содержать несколько предложений, если мастер, например, не ответил вовремя и получил следующий оффер. Мастер может открыть карточку каждой заявки, где указаны адрес (город, район, улица, дом), категория работы, описание проблемы (без контактов клиента до назначения). В карточке оффера мастер может нажать «Взять заказ» (соглашается на выполнение) или «Отказаться». Если мастер соглашается, заявка мгновенно получает статус ASSIGNED на него, остальные офферы отзываются. Если мастер отказывается или не успел ответить (SLA 2 минуты истекло), заявка продолжит поиск другого кандидата. Отказ/таймаут оффера не штрафует мастера, кроме случая гарантийных заявок (см. §7.4, §10).
•	Активный заказ: У мастера в меню отображается раздел «Активный заказ» (если у него назначена заявка в статусе ASSIGNED или далее). Мастер переходит в карточку активного заказа, где видит детали: адрес, контакты клиента (телефон клиента становится видимым только после назначения заказа мастеру), описание проблемы, прикрепленные фото/файлы, выбранный клиентом слот визита. Мастер последовательно обновляет этапы выполнения:
•	По выезду к клиенту нажимает кнопку «В пути» – заказ переходит в статус EN_ROUTE. Клиент (через оператора) и админ могут видеть, что мастер выехал.
•	По прибытии и началу работы мастер нажимает «Работаю» – статус WORKING.
•	По завершении работы мастер нажимает «Закрыть заказ» – бот запросит у него сумму, полученную с клиента за работу. Сумма должна быть > 0 и вводится в формате целых или десятичного числа с точкой (регулярное выражение проверки: ^\d{1,7}(\.\d{1,2})?$); при неверном формате выдаст ошибку («Неверный формат суммы. Пример: 3500 или 4999.99»). После ввода суммы мастер обязательно загружает фото или скан акта выполненных работ (формат – фото либо документ PDF). Акт подтверждает выполнение услуги и сумму. Как только акт загружен, заказ переходит в статус PAYMENT (ожидает оплату комиссии мастером), и для данного заказа создается запись комиссии (см. §9).
•	Оплата комиссии: После закрытия заказа для мастера автоматически формируется обязательство по оплате комиссии сервису. В мастер-боте появляется раздел «Финансы» (или обновляется, если уже был). Подробнее о комиссиях – в §9, но процесс для мастера: он видит сумму комиссии, дедлайн оплаты, реквизиты для перевода и может отправить чек. После полной оплаты комиссия помечается оплаченной, а заказ переводится в статус CLOSED (окончательно закрыт).
9. Финансы и комиссии
Монетизация сервиса осуществляется через комиссию, которую мастер платит с каждого заказа. Ниже описана логика расчёта и процесс оплаты комиссии:
•	Базовая ставка комиссии: По умолчанию комиссия составляет 50% от суммы заказа. Однако если мастер за последнюю неделю имеет высокий средний чек (>= 7000 руб), для поощрения устанавливается пониженная ставка 40%. Решение о применении 40% ставки принимается при закрытии заказа (т.е. проверяется показатель avg_week_check мастера на момент закрытия).
•	Расчет комиссии: После того, как мастер вводит сумму заказа, система вычисляет commission = round(order.total_sum * rate, 2) – то есть процент по ставке, округленный до 2 знаков. Например, если сумма заказа 3000 и ставка 50%, комиссия = 1500.00 руб.
•	Создание записи комиссии: В базе (таблица commissions) создается запись с привязкой к заказу и мастеру, суммой комиссии и ставкой. Также делается снапшот реквизитов владельца (того, кто принимает оплату) на момент создания комиссии: в поле snapshot хранятся способы оплаты, номер карты (только последние 4 цифры), телефон для СБП (замаскированный) и прочая информация, необходимая для идентификации платежа. Например, комментарий к платежу для мастера может автоматически формироваться как «Комиссия №12345 от Иванов И.И.» – где №12345 – номер комиссии, а Иванов И.И. – владелец/админ.
•	Дедлайн оплаты: Мастер должен оплатить комиссию в течение 3 часов с момента создания комиссии. Дата и время дедлайна рассчитываются при создании (например, заказ закрыт в 15:00 – дедлайн оплаты комиссии 18:00 того же дня). Этот дедлайн сохраняется в комиссии (deadline_at). Просрочка платежа ведет к санкциям (см. ниже).
•	Интерфейс мастера (финансы): В мастер-боте комиссии разделены на три группы: «Ожидают оплаты», «Оплаченные», «Просроченные». Мастер может открыть карточку конкретной комиссии, где указана сумма, ставка, дедлайн, а также кнопки:
•	Реквизиты: показать актуальные реквизиты для оплаты (на основе snapshot и текущих настроек владельца – карта, СБП, QR-код и т.п.).
•	Отправить чек: позволяет загрузить квитанцию/скриншот оплаты (фото или PDF). Чеки сохраняются и доступны админам.
•	Я оплатил: мастер нажимает после фактической отправки денег. Этот статус фиксируется как reported – время нажатия сохраняется (paid_reported_at), и комиссия ожидает подтверждения от администратора.
•	Интерфейс администратора (финансы): Админ-бот имеет раздел управления комиссиями. Админ видит поступившие от мастеров чеки и заявки на подтверждение оплаты. Для каждой комиссии доступны действия:
•	Посмотреть чеки: открыть загруженные файлы оплаты.
•	Подтвердить оплату: отмечает комиссию как is_paid=true, сохраняет время подтверждения (paid_approved_at) и фактически полученную сумму (paid_amount, может отличаться, если мастер, например, оплатил частями или с округлением). При подтверждении связанный заказ (если находился в статусе PAYMENT) переводится в CLOSED.
•	Отклонить оплату: если чек некорректен или оплата не поступила – комиссия остается неоплаченной, мастер уведомляется с причиной отказа. Он может повторно отправить чек.
•	Заблокировать мастера: в случае явных нарушений (например, подделка чека) админ может сразу заблокировать мастера из этого интерфейса. При блокировке мастер лишается доступа к новым заказам (is_active=false).
•	Просрочка оплаты: Если 3 часа прошли, а мастер не нажал «Я оплатил» и не была подтверждена оплата, комиссия считается просроченной. Система автоматически блокирует мастера (ставит is_active=false, reason "overdue commission") и отправляет тревожное уведомление администраторам о просроченной комиссии (алерт). Разблокировать мастера после просрочки можно только вручную администратором (после разбирательства и оплаты).
10. Гарантийные заявки
Гарантийная заявка – это повторный выезд мастера по уже выполненному заказу, если клиент выявил проблему/недочет в рамках гарантийного обязательства сервиса. Особенности работы с гарантиями:
•	Создание гарантии: Гарантийный заказ создается администратором из карточки исходного закрытого заказа (например, кнопкой «Создать гарантию»). В новой заявке поля адреса, контактов и описания копируются из исходного заказа, админ может добавить комментарий о сути рекламации. Такой заказ получает тип GUARANTEE и пометку «ЗАЯВКА ПО ГАРАНТИИ» в интерфейсе.
•	Company payment: Для клиента гарантийный визит бесплатный, а мастер получает фиксированную оплату от компании. В систему это отражено полем company_payment = 2500 (например, 2500 руб. – стандартная выплата мастеру за гарантийный случай). Поскольку клиент не платит, поле total_sum = 0, и комиссия с мастера не взимается (rate=0, комиссия=0).
•	Распределение гарантийной заявки: Автоматический поиск мастера выполняется, но с приоритетом исходного мастера: алгоритм ставит предыдущего исполнителя первым кандидатом (см. §7.4). Предыдущему мастеру отправляется оффер с пометкой, что это его гарантийный выезд. Если он принимает – заявка назначается ему, если отказывается или не отвечает – система автоматически блокирует этого мастера (так как отказ от гарантийного выезда недопустим). После отказа заявка переходит в обычный режим распределения, и другим мастерам рассылаются офферы по стандартным правилам.
•	Особенности выполнения: Процесс для мастера аналогичен обычному заказу (статусы EN_ROUTE, WORKING, CLOSED), но сумма не вводится (так как оплаты нет) и акт все равно прикрепляется. Комиссия не создается, однако админ может учитывать такой заказ отдельно (например, для выплат мастеру из бюджета компании).
11. Оценки и отзывы клиентов
После завершения заказа (перевод в CLOSED) сервис собирает обратную связь от клиента о качестве работы мастера:
•	Менеджер/админ связывается с клиентом (вне бота, например, по телефону или другим каналам) и узнает оценку работы мастера по шкале 1–5, а также текстовый отзыв (комментарий). В интерфейсе админ-бота в карточке закрытого заказа есть возможность сохранить оценку и отзыв.
•	Сохраненная оценка участвует в вычислении среднего рейтинга мастера (rating_avg), который хранится в профиле мастера и автоматически пересчитывается после каждого нового отзыва. Этот рейтинг используется в алгоритме ранжирования при распределении (§7.3) – более высокий средний рейтинг повышает шансы мастера получать заказы.
•	Текстовые отзывы сохраняются и могут просматриваться администраторами. Мастеру в бот приходит уведомление с информацией о полученной оценке и самим отзывом (например: «Ваш заказ №12345 закрыт. Клиент оценил работу на 4 из 5 и оставил отзыв: “Приехал вовремя, все починил, но не убрал мусор.”»). Это помогает мастерам получать обратную связь и повышать качество услуг.
12. Реферальная система
Для поощрения привлечения новых мастеров сервис поддерживает двухуровневую реферальную программу:
•	Каждый мастер получает уникальный реферальный код, которым он может поделиться с потенциальными новыми мастерами. Когда новый мастер проходит онбординг, его могут спросить «Есть ли реферальный код?» – если он введет код действующего мастера, устанавливается связь «реферер – реферал».
•	Двухуровневая схема вознаграждений: Если мастер A пригласил мастера B (ввел его код), то A получает 10% от всех комиссий, уплаченных B. Если затем мастер B пригласил мастера C, то A (будучи «прародителем» уровня 2) получает 5% от комиссий C. Соответственно:
•	Уровень 1: 10% от комиссии приглашенного мастера первого уровня.
•	Уровень 2: 5% от комиссии мастера второго уровня.
•	Начисления фиксируются в таблице referral_rewards: каждая оплата комиссии генерирует записи вознаграждений для рефереров 1-го и 2-го уровня (с указанием order_id, кто платил комиссию, кто получил награду, уровень, сумма). Эти суммы позже используются админами для выплат пригласившим мастерам (отдельно, вне бота).
•	В интерфейсе мастер-бота может быть раздел «Реферальная программа», где мастер видит свой код, количество приглашенных и сумму заработанных бонусов. В админ-боте раздел отчетов позволяет выгружать данные по рефералке (см. §16).
13. Админ-бот (Логист/Админ): функции и UI
Админ-бот объединяет функциональность логиста и администратора (права зависят от роли пользователя, см. §1). Ниже перечислены основные разделы и возможности:
•	Очередь заявок: Главный экран для логиста/админа – список новых и активных заявок с фильтрами. Можно отфильтровать по городу, категории, статусу, назначенному мастеру, дате создания/слоту. Отображаются только заявки в городах, закрепленных за пользователем. Список показывает краткую информацию (ID, адрес, категорию, статус, назначенный мастер или «в поиске»). Логист может выбрать заявку, чтобы открыть карточку заказа: там полная информация – описание, вложения (можно скачать фото/файлы), контакты клиента, назначенный мастер (если есть) и история статусов. В карточке доступны действия:
•	Назначить мастера (авто): запустить алгоритм автоматического распределения вручную для данного заказа (полезно, если ранее авто-поиск был отложен или если нужно повторно запустить). По нажатию заявки присваивается статус SEARCHING и сразу выполняется распределение по правилам §7 (2 раунда и т.д.).
•	Назначить мастера (вручную): открыть список доступных мастеров для ручного выбора. Админ-бот покажет мастеров соответствующей категории и городу (и при наличии фильтра по району – возможно сначала всех, с возможностью фильтрации?). В списке видно, кто на смене (отметка ✅ или ❌), рейтинг, наличие авто. По умолчанию мастера вне смены тоже отображаются (вдруг нужно срочно назначить). Если логист пытается назначить мастера, который не на смене, бот потребует подтверждения («Мастер сейчас не на смене, вы уверены?»). После выбора мастера ему отправляется оффер через бот (как обычно в «Новых» у мастера). Если мастер примет, заказ станет ASSIGNED; если откажется – заявка вернется в очередь.
•	Вернуть в поиск: доступно, если заказ был назначен мастеру, но нужно снова искать другого (например, мастер по какой-то причине отменен). Эта команда снимает назначение (возвращает статус SEARCHING) и может сразу запускать новый цикл авто-распределения или оставить на ручной контроль.
•	Отменить заявку: позволяет установить статус CANCELED с указанием причины отмены (например, клиент передумал, дублирование, и т.д.). Отмена фиксируется в истории, мастер (если был назначен) уведомляется об отмене.
•	Новая заявка: Отдельный раздел для создания заявки вручную (аналог процесса §6). Логист/админ по шагам заполняет данные клиента, адрес, описание, фото, слот. После подтверждения данных бот предлагает сразу распределить заявку: можно выбрать авто-назначение (поиск мастера автоматически) или ручное (сразу открыть список мастеров). Если выбран ручной и мастер – сразу заявка назначается. Если авто – заявка уходит в SEARCHING и алгоритм §7 отправляет оффер мастеру. (Примечание: если адрес без района – авто-распределение не запустится, заявка просто создастся и останется логисту для ручного назначения.)
•	Гарантия: В карточке любого закрытого заказа админ видит кнопку «Гарантийный выезд» (если подобный уже не создан). При нажатии формируется гарантийная заявка (как описано в §10) с типом GUARANTEE, привязанная к исходному заказу. Логист далее может распределить ее (но авто-алгоритм сам сначала предложит исходному мастеру). Гарантийные заявки отмечаются особо, чтобы не перепутать с обычными.
•	Мастера (управление персоналом): Раздел для просмотра и модерации анкет мастеров:
•	Списки мастеров группируются по статусу: «На модерации», «Одобренные», «Заблокированные». В каждом списке можно листать всех мастеров, фильтровать по категориям/городу.
•	Карточка мастера показывает данные анкеты: ФИО, телефон, город, районы, наличие авто, навыки, средний чек, рейтинг, текущее состояние (на смене/нет, на перерыве/нет), количество активных заказов, реквизиты для выплат, реферальный код, пригласивший мастер и т.д. Также видны загруженные документы: паспорт, селфи, чеки оплат.
•	Действия с мастером: Одобрить (для новых – устанавливает verified=true, отправляет уведомление мастеру о допуске к работе), Отклонить (для анкеты на модерации – запросить причину отказа, уведомить мастера), Заблокировать (для действующего мастера – сразу отключает его, попросит указать причину блокировки, напр. системное нарушение), Разблокировать, Удалить (удаляет профиль мастера, если он, например, никогда не работал и был ошибочно зарегистрирован – требуется подтверждение).
•	Просмотр документов: возможность просматривать паспорт, селфи, и все файлы, которые мастер загружал (по file_id скачиваются с серверов Telegram).
•	Лимит активных заказов: возможность изменить персонально лимит max_active_orders для конкретного мастера (по умолчанию 5, но можно поставить 1-2 для новичков или снять ограничение для самых надежных).
•	Модерация анкет: Подраздел (может быть объединен с мастерами) для рассмотрения новых заявок мастеров. Показывает всех мастеров со статусом "на модерации". Админ просматривает анкету, документы и либо одобряет, либо отклоняет анкету. При отклонении указывает причину (например, нечитаем паспорт, некорректные данные) – мастер получает уведомление и может повторно отправить данные или обращается в поддержку.
•	Финансы (комиссии): Раздел управления комиссиями и реквизитами:
•	Ожидают оплаты / Оплаченные / Просроченные комиссии: списки соответствующих записей комиссии. Админ может открывать каждую для деталей. В карточке комиссии указано: ID комиссии, заказ, мастер (ФИО, телефон, ID), сумма, ставка, дедлайн, отметки о репорте оплаты, ссылки на чеки, статус. Доступны действия: Открыть чеки (посмотреть прикрепленные файлы), Подтвердить оплату, Отклонить (с указанием причины), Блокировать мастера (если, например, подозрение на мошенничество).
•	Настройки реквизитов владельца: глобальный админ может изменить реквизиты, на которые мастера платят комиссии – например, номер карты, телефон для СБП, загрузить новый QR-код для оплаты, изменить текст комментария к платежу. Эти реквизиты хранятся в настройках и рассылаются мастерам. Есть возможность разослать обновленные реквизиты всем мастерам с неоплаченными комиссиями (кнопка «Обновить реквизиты и уведомить»), чтобы они видели актуальную информацию.
•	Рассылка реквизитов: если реквизиты поменялись, бот может автоматически разослать сообщение мастерам, у кого ожидается оплата, с новыми данными (вместо ручного оповещения).
•	Отчёты: Админ может выгрузить данные по ключевым сущностям в CSV или XLSX файлы:
•	Заказы: Полный список заказов или по фильтру (например, за период) в формате CSV/XLSX (см. §16 для колонок).
•	Комиссии: Отчёт по комиссиям (все или за период).
•	Рефералка: Отчет по реферальным вознаграждениям. Выгрузки применяются в управленческих целях (например, сверка финансов). CSV файлы генерируются в кодировке UTF-8 с BOM для корректного открытия в Excel; разделитель ;. В XLSX делаются отдельные листы по типам данных.
•	Настройки: Панель настройки системы (доступно глобальному администратору, частично городским):
•	Рабочие часы/слоты: настройка глобальных переменных времени работы (например, изменить время начала/окончания дня, длительность слотов и т.д.).
•	SLA/раунды/эскалации: настройка параметров авто-распределения – длительность SLA секунд, число раундов, таймаут до эскалации на админа.
•	Лимит активных заказов: глобальный лимит (по умолчанию 5).
•	FAQ и поддержка: управление содержимым справочного раздела для мастеров (часто задаваемые вопросы, инструкции) и контактной информации службы поддержки. Мастера могут видеть это в разделе «База знаний» своего бота.
•	Каналы логов: настройка Telegram-каналов (или чатов) для отправки системных уведомлений: алертов, логов распределения, отчетов. Админ указывает chat_id для канала оповещений, логов, отчётов.
•	Доступ и персонал (только глобальный админ): Интерфейс управления учетными записями персонала:
•	Список всех сотрудников (глобальные админы, городские админы, логисты) с указанием их ролей, ФИО, телефона, активности.
•	Выдать новый код доступа: выбор роли и городов (для городских), генерация одноразового кода, который сообщается новому сотруднику.
•	Отозвать код: если код был выдан, но не использован, его можно отменить.
•	Редактировать привязанные города у городских ролей.
•	Деактивировать учетную запись сотрудника (без удаления, например, уволен – is_active=false, чтобы не мог пользоваться ботом).
14. Настройки (.env / конфигурация)
Основные настройки системы хранятся либо в файле окружения (.env), либо в админ-панели (раздел «Настройки», см. §13). Ниже перечислены ключевые параметры:
•	Токены Telegram-ботов: MASTER_BOT_TOKEN и ADMIN_BOT_TOKEN – токены, полученные от BotFather для мастер-бота и админ-бота соответственно. Каждый бот работает под своим токеном.
•	Роли (Telegram ID): Предварительно, для запуска системы, в .env можно указать списки Telegram ID, которые имеют права администратора:
•	GLOBAL_ADMINS_TG_IDS – JSON-массив ID для глобальных админов,
•	ADMINS_TG_IDS – для городских админов,
•	LOGISTIC_TG_IDS – для логистов.
Эти списки используются при первом старте, но далее управление персоналом происходит через интерфейс админ-бота (см. §13).
•	База данных (PostgreSQL): строка подключения DATABASE_URL формата postgresql+asyncpg://user:pass@host:port/dbname. В системе одна общая база, где все схемы объединены (мастера, заказы, и т.д. в одних таблицах с полем города для фильтрации).
•	Telegram-каналы/уведомления: ALERTS_CHANNEL_ID, LOGS_CHANNEL_ID, REPORTS_CHANNEL_ID – ID чатов или каналов, куда будут отправляться системные сообщения:
•	Алерты (критические уведомления, ошибки) – например, просроченная комиссия, эскалация заявки.
•	Логи распределения (строки от планировщика о ходе авто-назначения, см. Приложение A).
•	Регулярные отчёты или heartbeat (см. §18).
•	Параметры распределения и времени:
•	SLA_SECONDS (по умолчанию 120),
•	DISTR_ROUNDS (количество раундов авто-офферов, по умолчанию 2),
•	ESCALATE_TO_ADMIN_AFTER_MIN (через сколько минут после неудачи авто назначения оповещать администратора; напр. 10),
•	DEFAULT_MAX_ACTIVE_ORDERS (лимит активных заказов на мастера, по умолчанию 5),
•	WORKDAY_START и WORKDAY_END (например, 10:00 и 20:00).
Эти значения могут быть изменены админом через настройки (кроме, возможно, WORKDAY, если это глобально зафиксировано).
•	Геоданные: Настройки подключения внешних API отсутствуют (геокодирование не используется). Все необходимые координаты адресов берутся из локальной базы, которая наполняется заранее (см. Приложение D, миграции).
•	Реквизиты владельца: Хранятся данные для оплаты комиссий: какие методы оплаты включены (карта, СБП, др.), номер карты, телефон СБП, QR-код файла, текст комментария к платежу (см. §9). Эти данные редактируются только глобальным админом и рассылаются мастерам, когда изменяются.
15. Валидация вводимых данных и обработка ошибок
Система должна тщательно проверять пользовательский ввод (как от админов при создании заявок, так и от мастеров при онбординге и закрытии заказов). Ниже перечислены основные правила валидации и соответствующие сообщения об ошибках:
•	Телефон (клиента и мастера): При вводе проверяется формат – должно быть +7XXXXXXXXXX (11 цифр) или 8XXXXXXXXXX (10 цифр после 8). Любой другой формат считается недопустимым. При ошибке бот отвечает: «Неверный формат телефона. Пример: +7XXXXXXXXXX или 8XXXXXXXXXX». При сохранении номера, если он начинался с 8, он заменяется на +7.
•	ФИО мастера: Каждое из полей Фамилия, Имя, Отчество должно содержать от 2 до 30 символов, допускаются только буквы кириллицы, пробелы и дефисы. Никаких цифр или латиницы. При несоответствии – ошибка «ФИО может содержать только буквы русского алфавита, пробел или дефис, от 2 до 30 символов».
•	Поиск города: При вводе города на этапе создания заявки (или в анкете мастера) бот сопоставляет введенное с доступным списком. Если найдено несколько схожих вариантов, бот предлагает уточнить (список с номерами). Город вне списка – ошибка «Выберите город из списка предложенных».
•	Районы: Если для города есть справочник районов, они выводятся кнопками. Если пользователь не выбрал район из кнопок (например, ввел текстом что-то) – бот напомнит: «Пожалуйста, используйте кнопки ниже для выбора района». При отсутствии справочника районов бот может пропустить этот шаг или предложить сразу ввод улицы.
•	Улица: При поиске улицы с RapidFuzz бот отображает результаты. Если пользователь не нажал предложенную кнопку, а ввел вручную, строка проверяется на длину 2–50 символов. Также предусмотрена кнопка «Нет улицы». Если ввод не проходит критерии – сообщение «Название улицы должно содержать от 2 до 50 символов или выберите вариант из списка».
•	Дом: Проверяется, что длина от 1 до 10, символы цифры или буквы (латинские/кириллические?), допускаются «/» (слеш), «к» и «с» (корпус, строение). Примеры допустимых: "10", "10А", "14/3", "25к2". При неверном формате – «Номер дома может содержать цифры, буквы, /, к, с, до 10 символов».
•	Описание проблемы: Минимум 10 символов, максимум 500. Если короче – «Опишите проблему подробнее (не менее 10 символов)», если слишком длинно – «Описание слишком длинное, сократите до 500 символов».
•	Сумма заказа (мастер закрывает заказ): Проверяется регулярным выражением на числовой формат с опциональной копеечной частью, и должно быть больше 0. При ошибке: «Неверный формат суммы. Пример: 3500 или 4999.99».
•	Медиа-файлы: Бот должен корректно обрабатывать типы отправляемых файлов. Ожидаемые типы:
•	Паспорт, акт, чек могут быть отправлены как фото (тип InputPhoto) либо как документ (PDF или изображение как файл). Видео или голосовые сообщения не принимаются вместо требуемых документов. Если пользователь попытался отправить недопустимый тип, бот отвечает: «Недопустимый формат файла, отправьте фото или PDF-документ.».
•	При получении фото документов бот сохраняет file_id и не обязательно скачивает файл на диск. Телеграм позволяет повторно использовать file_id для отправки файлов при необходимости[1].
•	Ошибки использования UI: В случаях, когда бот ожидает, что пользователь нажмет одну из предоставленных кнопок (например, выбор района, подтверждение чего-либо) и получает текстовое сообщение вместо этого, выдается стандартный ответ: «Пожалуйста, используйте кнопки ниже». Это касается любых меню с inline-кнопками.
•	Таймаут FSM: Если пользователь слишком долго (например, >5 минут) не отвечает в процессе многошагового диалога (Finite State Machine), сессия сбрасывается. Бот отправляет сообщение: «Сессия истекла. Нажмите /start для начала» – после чего пользователь может начать заново командой /start или с главного меню. В реализации FSM следует предусмотреть автоматическое завершение состояния по таймауту.
(Примечание: FSM – конечный автомат состояний – используется для диалогов онбординга мастера, создания заявки и пр. Он помогает пошагово запрашивать данные. Необходимо настроить корректный выход из состояний по /cancel или по истечении времени, чтобы бот не зависал.)
16. Выгрузки и отчёты (CSV/XLSX)
Система предусматривает экспорт данных в формат CSV и XLSX для анализа и отчетности. Общие требования к формату экспорта:
- CSV: Кодировка UTF-8 с BOM (Byte Order Mark) для совместимости с Excel; разделитель – точка с запятой ;. Даты/время представлены в формате ISO 8601 (UTC). Числовые значения – с десятичной точкой (принятый разделитель в CSV). Логические (булевы) значения – true/false. Первая строка – заголовки колонок.
- XLSX: Генерируется Excel-файл с отдельными листами для разных типов данных, если выгружается пакетно (например, один файл со всеми отчетами). Наименования листов: orders, commissions, ref_rewards и т.д. Заголовки колонок такие же, как в CSV. Даты можно записывать как текст в ISO 8601 во избежание проблем с часовыми поясами.
Ниже перечислены состав колонок для основных отчетов (порядок колонок строгий, как указано):
•	Заказы (orders_*.csv):
order_id; created_at_utc; closed_at_utc; city; district; street; house; lat; lon; category; status; type; timeslot_start_utc; timeslot_end_utc; late_visit; company_payment; total_sum; user_name; user_phone; master_name; master_phone; cancel_reason
•	Комментарий: late_visit – признак опоздания мастера (булево, если реализовано измерение опоздания относительно слота), company_payment – сумма выплаты мастеру от компании (для гарантий, например 2500), cancel_reason – текст причины отмены если статус CANCELED, иначе пусто.
•	Комиссии (commissions_*.csv):
commission_id; order_id; master_id; master_name; master_phone; amount; rate; created_at_utc; deadline_at_utc; paid_reported_at_utc; paid_approved_at_utc; paid_amount; is_paid; has_checks; snapshot_methods; snapshot_card_number_last4; snapshot_sbp_phone_masked
•	Комментарий: has_checks – есть ли прикрепленные файлы квитанций (true/false), snapshot_methods – какие методы оплаты были доступны (например "card,sbp"), номер карты маскируется кроме последних 4 цифр, телефон СБП частично скрыт (например, +7xxxxxx1234).
•	Реферальные вознаграждения (ref_rewards_*.csv):
reward_id; master_id; order_id; level; amount; created_at_utc
•	Здесь master_id – кому начислено вознаграждение, level – уровень (1 или 2), order_id – за какой заказ (соответствует комиссии приглашенного), amount – сумма выплаты.
(При генерации XLSX эти же поля используются на соответствующих листах. Важно, чтобы формат CSV был строго соблюден для удобства парсинга.)
17. База данных – ключевые таблицы
Ниже приведен обзор основной схемы базы данных PostgreSQL для данной системы. Все таблицы связаны в рамках одной схемы; используется SQLAlchemy/Alembic для миграций.
•	masters: информация о мастерах (ID, ФИО, телефон, город_id, список районов (может храниться как текст или в связанной таблице master_districts), наличие авто, навыки (можно как bitmask или отдельная таблица master_skills), флаги: verified, is_active, is_deleted, is_on_shift, on_break_until, рейтинг, средний чек и т.п., реквизиты оплаты, реферальный код, id реферера).
•	orders: заявки/заказы. Ключевые поля: order_id, тип (type: NORMAL или GUARANTEE), статус, город_id, район_id, улица_id, дом (текст), координаты lat, lon, категория, временное окно (timeslot_start, timeslot_end в UTC), late_visit (булево, если мастер опоздал), company_payment, total_sum, user_name, user_phone, master_id (если назначен), cancel_reason.
•	order_status_history: история изменений статусов заказов. Поля: id, order_id, предыдущий статус, новый статус, timestamp, changed_by (id пользователя или мастера, или системой), reason.
•	offers: офферы заявок мастерам. Поля: order_id, master_id, created_at, статус (отправлен/принят/отклонён/просрочен), deadline. Уникальный ключ (order_id, master_id) чтобы один мастер не получил один оффер дважды. Это помогает в идемпотентности рассылки офферов.
•	attachments: вложения (файлы). Поля: id, order_id (если файл к заказу), commission_id (если файл к комиссии), master_id (если личный документ мастера), тип (например, passport, selfie, act, commission_check, photo, doc и т.д.), telegram_file_id (строка идентификатора файла в Telegram для последующего использования). Дополнительно может храниться original_file_name, file_size, mime_type для удобства.
•	commissions: комиссии мастеров. Поля: commission_id, order_id, master_id, сумма (amount), ставка (rate), created_at, deadline_at, paid_reported_at, paid_approved_at, paid_amount, is_paid (булево), и snapshot реквизитов: хранится JSON с полями (methods, card_last4, sbp_phone_masked, owner_name и др.). Также has_checks (наличие чеков). Индексы по is_paid, deadline_at для быстрой выборки просроченных и неоплаченных.
•	referrals: связи рефералов. Поля: id, ref_code, master_id (чей код), invited_master_id (кто зарегистрировался). Можно хранить level, но уровень 1 или 2 вычисляется через цепочку.
•	referral_rewards: начисления рефералки. Поля: reward_id, master_id (кому бонус), from_master_id (кто совершил заказ, по которому бонус), order_id, level, amount, created_at.
•	cities, districts, streets: справочники географии. Связаны иерархически: districts имеют city_id, streets имеют district_id (или city_id, если город без районов). Хранят названия и координаты (lat, lon) центроидов.
•	settings: глобальные настройки (например, JSON поля для реквизитов, может быть таблица key-value).
•	staff_users: учетные записи персонала. Поля: staff_id, telegram_id, роль (GLOBAL_ADMIN, CITY_ADMIN, LOGIST), ФИО, телефон, is_active, date_added.
•	staff_cities: привязка городов для городских админов и логистов. Поля: staff_id, city_id (может быть много записей, если сотрудник ведет несколько городов).
Индексы и производительность:
- Для таблицы orders важно иметь индекс по (status, city_id) чтобы быстро находить активные заявки по городам для распределения и отображения очереди.
- Для masters – индекс по (is_on_shift, verified) чтобы быстро выбирать доступных кандидатов.
- Для offers – уникальный индекс (order_id, master_id) для предотвращения дубликатов офферов. Индекс по order_id для быстрого получения всех офферов заявки (когда нужно отозвать/очистить при назначении).
- Для commissions – индекс по (is_paid, deadline_at) для выборки неоплаченных и проверки просрочек.
- Важна целостность: при назначении заказа мастеру нужно транзакционно изменить order (назначить master_id, статус ASSIGNED) и зафиксировать оффер как принятый, отказ остальных – эти операции должны быть атомарными.
18. Мониторинг, уведомления и эксплуатация
Чтобы система работала устойчиво, предусмотрены механизмы мониторинга состояния ботов и автоматических оповещений об ошибках:
•	Heartbeat ботов: Каждый бот (мастер-бот и админ-бот) должен периодически (раз в 60 секунд) отправлять сервисное сообщение в канал логов (если настроен LOGS_CHANNEL_ID). Например, сообщение «[MASTER_BOT] Heartbeat OK» с отметкой времени. Это нужно, чтобы убедиться, что бот запущен и не завис. Отсутствие heartbeat более заданного времени сигнализирует о проблеме.
•	Алерты: В канал алертов (ALERTS_CHANNEL_ID) отправляются сообщения о важных событиях:
•	Любая ошибка выполнения кода (необработанное исключение, сбой) – бот отлавливает глобально и шлет уведомление с текстом ошибки и стектрейсом (чтобы разработчики могли быстро реагировать).
•	Просрочка комиссии: когда комиссия становится просроченной (мастер заблокирован) – автоматическое уведомление, указать мастера, сумму, заказ.
•	Пустая очередь мастеров: если для новой заявки не нашлось ни одного кандидата (например, никто не на смене или нет мастеров с нужным навыком) – сигнал, что в городе проблема с наличием мастеров.
•	Эскалации: если заявка ушла на эскалацию (после 2 раундов не назначена) – уведомление, чтобы админ взял в работу.
•	No_district: если заявка создана без района и авто-распределение пропущено – уведомление логисту, чтобы срочно назначил вручную.
•	Единственный инстанс каждого бота: Необходимо запускать только по одному экземпляру мастер-бота и админ-бота. Если случайно запущены два экземпляра с одинаковым токеном (например, при деплое) – при параллельном использовании getUpdates от Telegram возникнет ошибка 409 Conflict: terminated by other getUpdates request[2]. Боты должны уметь определять такой конфликт: если при попытке чтения обновлений получена ошибка 409, бот должен логгировать это и завершаться, либо механизм оркестрации должен гарантировать, что второй инстанс не запускается. Таким образом, соблюдается требование Telegram: только один процесс опрашивает обновления бота одновременно[2]. (Альтернативно, можно использовать вебхуки для масштабирования, но в данной системе выбран long polling).
•	Бэкапы базы данных: Должно быть настроено регулярное резервное копирование PostgreSQL. Например, daily pg_dump -Fc (сжатый custom формат) каждый день ночью; хранить минимум 7 последних дней резервных копий (чтобы откатиться в случае сбоя). Желательно автоматизировать проверку успешности бэкапа и уведомлять, если бэкап не создавался.
19. Нефункциональные требования (NFR)
Помимо функциональности, система должна удовлетворять следующим нефункциональным требованиям:
•	Производительность: Автораспределение должно отрабатывать быстро. Цикл распределения (обработка всех активных заявок в SEARCHING) должен выполняться не более 1 секунды при нагрузке до 100 активных заявок одновременно. Это достигается оптимизированными запросами (отбор кандидатов по индексам) и асинхронной обработкой. Интерфейс ботов должен быстро реагировать на действия пользователей (в пределах пары секунд на операцию).
•	Идемпотентность и корректность: Система должна избегать дублирования или потери данных. Например, отправка оффера одному и тому же мастеру повторно – недопустима (гарантируется уникальным индексом offers). Назначение заказа мастеру должно быть атомарным – исключая ситуацию, когда два мастера примут один заказ одновременно. История статусов и логирование действий обеспечивает восстановление хода событий при разборе инцидентов.
•	Надёжность: Все важные действия (назначение, закрытие заказа, оплата комиссии) должны выполняться в транзакциях с проверкой актуальности данных (optimistic locking или FOR UPDATE при необходимости). Бот должен корректно работать с перебоями связи: в случае недоступности Telegram API или базы данных – делать повторные попытки, сообщать об ошибках.
•	Безопасность данных: Персональные данные клиентов и мастеров (имена, телефоны, фото документов) обрабатываются согласно согласиям (см. Приложение B). Доступ разграничен ролями (мастер видит только свои заказы и минимальную информацию о клиенте; логисты/админы – только по своим городам). Телефон клиента отображается мастеру только после назначения на заказ. Номера банковских карт нигде не хранятся полностью (только маскированно), телефоны СБП – частично скрытые. Документы (паспорт, акт, чеки) не сохраняются на сторонних серверах, а хранятся как file_id Telegram – то есть фактически на серверах Telegram без возможности доступа по URL извне, только через бот. Это снижает риски утечки, так как даже в базе нет файлов, только ссылки на Telegram.
•	Масштабирование: При росте количества заявок и мастеров возможно запускать ботов на нескольких процессах/серверах, но нужно переходить на вебхуки, чтобы избежать 409 конфликта при poll. В текущей версии предполагается, что одна машина справится с нагрузкой (100+ одновременных активных заказов, что требует ~ сотен мастеров онлайн).
•	Логирование и отладка: Все ключевые события (создание заявок, назначение, статусы, оплаты) логируются либо в базу, либо в файлы/каналы. Это упрощает отладку и поддержку. Для разработчиков предусмотрен режим DEBUG, где можно включить расширенный лог команд бота, payloads и т.п. (только на тестовых контурах).
20. Приёмочное тестирование (UAT чек-лист)
Перед запуском системы необходимо пройтись по чек-листу, проверяя, что каждая функция работает в соответствии с ТЗ:
•	Создание заявки и распределение: Админ (или логист) создает новую заявку через бот. Проверить:
•	Если не выбран район, то заявка сразу отображается как требующая ручного назначения (флаг no_district), авто-распределение не стартует.
•	Если создано до 20:00, заявка сразу переходит в поиск мастера. Если после 20:00, получает статус DEFERRED и не должна быть видна мастерам до наступления 10:00 следующего дня. В 10:00 (локально для города) проверить, что статус автоматически сменился на SEARCHING и запущен поиск.
•	Если попытаться назначить вручную мастера, который не на смене, бот должен запросить подтверждение у логиста («Мастер не на смене, назначить всё равно?»). При согласии – оффер отправляется мастеру (он получит уведомление, хотя был оффлайн).
•	Алгоритм распределения: Смоделировать несколько заявок и мастеров:
•	Убедиться, что кандидаты подбираются строго по городу и району: мастер из другого города заявку не получает; если мастер выключил смену или на перерыве – его пропускают; если категория не подходит – тоже пропускают.
•	Проверить ранжирование: Создать 2-3 мастеров с разными параметрами (один с авто, другой без; разными рейтингами и средним чеком) – убедиться, что в логах распределения первым идет мастер с автомобилем, затем по avg_week_check, затем по рейтингу. Если у двух мастеров похожие показатели, наблюдать, что иногда порядок меняется из-за случайного фактора.
•	SLA 120 сек: Отправить оффер мастеру и не реагировать – убедиться, что через 2 минуты заказ автоматически предложится следующему кандидату (или эскалируется после 2 раундов). Проверить, что мастер после таймаута может все еще брать другие заказы и не считается назначенным.
•	no_district: Создать заявку без района – убедиться, что она сразу помечена как требующая ручного назначения (алерт логисту).
•	Эскалация: Смоделировать ситуацию, когда ни один мастер не принимает заказ (отключить всех или заставить отказать) – убедиться, что после 2 раундов заявка остается в SEARCHING, но авто-офферы прекращаются, и логист получает уведомление. Засечь 10 минут – приходит ли уведомление администратору о проблемной заявке.
•	Работа мастера:
•	Онбординг: Новый мастер проходит регистрацию. Ввести заведомо неправильные данные (неправильный телефон, слишком короткое имя, латиницу) – убедиться, что бот не дает продолжить без исправления. Загрузить не фото для паспорта (например, голосовое сообщение) – бот должен отклонить. После завершения анкеты убедиться, что мастер попал «на модерации» и не может включить смену, пока админ не одобрит.
•	Смена/перерыв: Одобрить мастера, зайти под ним: проверить, что по умолчанию он вне смены. Нажать «Включить смену» – статус изменился, админ-бот (для админа) может видеть его как онлайн. Нажать «Перерыв» – статус смены должен сняться на 2 часа. Проверить, что в течение этих 2 часов мастер не получает новых офферов. По окончании перерыва – автоматическое уведомление мастеру, что перерыв окончен (или просто переключение статуса). Также попробовать отключить смену вручную во время перерыва – бот должен разрешить (например, прервать перерыв).
•	Получение заказа: От имени логиста создать заявку, когда мастер на смене. Убедиться, что мастер-бот получает оффер в «Новых» – видит описание, адрес (без телефона клиента). Нажать «Взять заказ» – убедиться, что заказ переходит в ASSIGNED на этого мастера, другим мастерам оффер отзывается. Проверить, что мастер теперь видит заказ в «Активном».
•	Прохождение стадий: Мастер нажимает «В пути» – статус EN_ROUTE, логист/админ видит обновление в карточке заказа. Потом «Работаю» – статус WORKING. Затем «Закрыть»: мастер вводит сумму (попробовать ввести некорректно, напр. 0 или текст – бот откажет), ввести корректно, потом не прикреплять акт и нажать «Отправить» – бот не даст закрыть без акта. Прикрепить акт и подтвердить – заказ переходит в PAYMENT, комиссия создана.
•	Финансы мастера: После закрытия заказа мастер видит новую запись в «Ожидают оплаты». Проверить, что сумма и дедлайн верны (3 часа). Попробовать просрочить: ожидать 3 часа (или искусственно перевести время) – мастер должен быть блокирован, заказ не закрыт. Если прислать чек и нажать «Я оплатил» – админ должен увидеть комиссию в «Oжидают подтверждения». Под админом нажать «Подтвердить оплату» – комиссия пометилась оплаченной, мастер разблокирован (если был блок), заказ стал CLOSED. Мастер видит комиссию в «Оплаченные».
•	Гарантия: Закрыть обычный заказ, затем в админ-боте создать гарантийный. Проверить, что статус GUARANTEE, пометка видна. Исходному мастеру (если не заблокирован) уходит оффер. Отказаться от него – убедиться: мастер автоматически блокируется системой, в истории заявки это отражено, и другим мастерам пошли офферы. Назначить другому мастеру – тот выполняет, комиссия 0 руб (проверить, что мастер не видит комиссию за гарантию). Проверить, что company_payment=2500 отразилось в заказе и, например, в выгрузке orders.
•	Отчёты: Сгенерировать CSV и XLSX отчеты из админ-бота. Открыть в текстовом редакторе/Excel – убедиться, что кодировка правильная (русские буквы не испорчены), разделители верные, столбцы соответствуют описанию §16. Проверить, что даты в UTC соответствуют локальному времени + offset.
•	Оценки: После закрытия заказа в админ-боте добавить оценку и отзыв. Посмотреть, обновился ли рейтинг мастера (если было несколько оценок, средняя). Проверить, что мастер-бот получил уведомление с текстом отзыва.
•	Рефералка: Протестировать регистрацию нового мастера с вводом реферального кода существующего. Выполнить заказ новым мастером, оплатить комиссию. Проверить, что у пригласившего мастера появилась запись в referral_rewards с правильной суммой (10% или 5%). Также проверить выгрузку реферальных вознаграждений.
21. Структура UI-диалогов и callback-команд
Боты реализуют диалоговые процессы с помощью callback-кнопок и FSM состояний. Ниже приведены деревья команд и префиксов callback data для ориентировки (нормативно, чтобы разработчики/тестировщики могли сверяться):
21.1. Мастер-бот (префикс команд m:)
•	Главное меню: m:menu – отображает основные разделы (Новые заявки, Активный заказ, Смена, Финансы, Профиль, Реферальная программа, Статистика, База знаний).
•	Новые заявки: список офферов, странично
•	Команда: m:new:<page> – открыть страницу списка новых офферов.
•	Из списка: нажимая на заявку, команда m:new:card:<order_id> – открыть карточку оффера.
o	В карточке оффера кнопки:
o	m:new:acc:<order_id> – взять заказ (accept),
o	m:new:dec:<order_id> – отказаться (decline).
•	Активный заказ: если есть, префикс m:act
•	Команда m:act:card:<order_id> – открыть текущий активный заказ (может вызываться прямо m:act, если только 1 активный).
•	В карточке активного заказа этапы:
o	m:act:enr:<order_id> – установить статус En Route («В пути»).
o	m:act:wrk:<order_id> – установить статус Working («Работаю»).
o	m:act:cls:<order_id> – начать процесс закрытия (ввод суммы). Бот переходит в состояние ожидания суммы, затем документа.
o	После получения суммы и акта – завершается, но на UI можно предусмотреть финальную кнопку m:act:act:<order_id> – возможно подтверждение закрытия.
•	Смена: раздел управления сменой и перерывом.
•	m:sh:on – включить смену.
•	m:sh:off – выключить смену.
•	m:sh:brk – запрос перерыва. При нажатии бот присылает подтверждение: «Вы уверены взять перерыв 2 часа?» с кнопкой m:sh:brk:ok – подтвердить.
•	Финансы: раздел комиссий.
•	m:fin:aw – список ожидающих оплаты (awaiting).
•	m:fin:pd – список оплаченных.
•	m:fin:ov – список просроченных (overdue).
•	В списках: m:fin:cm:<cid> – открыть карточку комиссии.
o	В карточке комиссии:
o	m:fin:cm:pt:<cid> – показать реквизиты (pay to).
o	m:fin:cm:chk:<cid> – отправить чек (bot ждет документ/photo, после получения меняет has_checks).
o	m:fin:cm:ip:<cid> – нажать «Я оплатил» (payment reported, фиксируется время).
•	Профиль: m:pf – просмотр личных данных мастера (может включать реферальный код, рейтинг, и кнопки для редактирования некоторых данных).
•	Реферальная программа: m:rf – показать информацию о рефералах (код, приглашенные, заработано).
•	Статистика: m:st – данные по выполненным заказам, средний чек и т.д.
•	База знаний (FAQ): m:kb – справочные материалы. Админ может редактировать содержимое.
(Примечание: Редактирующие действия профиля могут иметь свои команды, например m:pf:edit_phone и т.п., но они опускаются для краткости.)
21.2. Админ-бот (префикс команд adm:)
•	Главное меню: adm:menu – показывает разделы: Очередь, Новая заявка, Мастера, Модерация, Финансы, Отчёты, Настройки, Персонал.
•	Очередь заявок:
•	adm:q – открыть список заявок (с последними фильтрами или все).
•	adm:q:flt – открыть меню фильтров (выбор города, категории, статуса и т.д.). После установки фильтров возвращается к списку.
•	adm:q:list:<page> – список заявок (постранично).
•	Нажатие на заявку: adm:q:card:<order_id> – открыть карточку заказа.
o	В карточке:
o	adm:q:as:<order_id> – подменю «Назначить» (открывает варианты авто/вручную).
o	adm:q:as:auto:<order_id> – запустить автораспределение (см. §7).
o	adm:q:as:man:<order_id>:<page> – открыть список мастеров вручную (постранично).
o	adm:q:as:pick:<order_id>:<master_id> – назначить выбранного мастера (отправляется оффер ему).
o	adm:q:ret:<order_id> – вернуть в поиск (если был назначен мастер, снимает назначение).
o	adm:q:cnl:<order_id> – отменить заявку (запросит подтверждение и причину).
o	adm:q:gar:<order_id> – (если заказ закрыт и не гарантийный) создать гарантийную заявку по нему.
•	Новая заявка:
•	adm:new:start – начать диалог создания новой заявки. Бот пошагово собирает:
o	категория (adm:new:cat:<category>),
o	адрес (adm:new:addr -> далее отдельный процесс с выбором города/района/улицы),
o	описание (adm:new:desc ожидает текст),
o	вложения (adm:new:att ожидает файлы, можно пропустить),
o	слот (adm:new:slot:<option> выбирается кнопками ASAP/сегодня/завтра).
•	После заполнения: adm:new:make – подтверждение создания (формируется заявка).
•	Затем бот предлагает способ назначения: adm:new:dispatch:auto|man.
o	Если auto: заявка создана и передана в авто-распределение.
o	Если man: бот откроет список мастеров, аналогично adm:q:as:man.
o	Выбор мастера: adm:new:pick:<order_id>:<mid> – назначить сразу.
•	Мастера:
•	adm:m:grp:<grp> – список групп мастеров, где <grp> может быть mod (на модерации), ok (одобренные), blk (заблокированные). Выводит первые несколько, или предлагает выбрать категорию фильтра.
•	adm:m:list:<grp>:<cat>:<page> – список мастеров группы с фильтром по категории (cat может быть all или конкретная).
•	Выбрать мастера: adm:m:card:<master_id> – открыть карточку мастера.
o	В карточке:
o	adm:m:ok – одобрить (если на модерации).
o	adm:m:rej – отклонить (если на модерации, бот запросит причину).
o	adm:m:blk – заблокировать (если активен, запрос причины).
o	adm:m:unb – разблокировать.
o	adm:m:del – удалить (если допустимо, попросит подтверждение).
o	adm:m:docs – просмотр документов (бот может отправить паспорт и селфи как сообщения).
o	adm:m:lim – изменить лимит активных заказов (бот запросит новое число, сохранит).
•	Модерация:
•	adm:mod:list:<page> – список анкет на модерации (аналог adm:m:grp:mod).
•	adm:mod:card:<master_id> – карточка конкретного мастера.
o	Действия: adm:mod:ok (одобрить), adm:mod:rej (отклонить с причиной) – аналогично действиям из раздела мастеров.
•	Финансы:
•	adm:f:aw – список ожидающих комиссий.
•	adm:f:pd – список оплаченных.
•	adm:f:ov – список просроченных.
•	Выбрать комиссию: adm:f:cm:<cid> – карточка комиссии.
o	Действия:
o	adm:f:cm:open – открыть чеки (бот вышлет файлы или ссылку на скачивание, либо перечислит названия).
o	adm:f:cm:ok – подтвердить оплату (спросит сумму фактически полученную, по умолчанию равна ожидаемой, можно отредактировать).
o	adm:f:cm:rej – отклонить оплату (спросит причину).
o	adm:f:cm:blk – заблокировать мастера (подтвердить).
•	Настройки реквизитов:
o	adm:f:set – показать текущие реквизиты владельца (методы, номера, комментарий).
o	adm:f:edit – изменить реквизиты (запустить процесс запроса новых данных: номер карты, телефон, загрузка QR и т.п.).
o	adm:f:bc – разослать актуальные реквизиты всем мастерам с неоплаченными комиссиями (broadcast).
•	Отчёты:
•	adm:r:o – сгенерировать отчет по заказам (например, за последний месяц или спросить период).
•	adm:r:c – отчет по комиссиям.
•	adm:r:rr – отчет по реферальным вознаграждениям. (Возможно, бот сразу формирует и отправляет файл, или сперва спрашивает период/фильтры.)
•	Настройки:
•	adm:s:wh – изменить рабочие часы/слоты (входит в подменю, позволяет редактировать начало/конец дня, окна).
•	adm:s:sla – настройки SLA/раундов/эскалаций (запрос числовых значений).
•	adm:s:lim – изменить глобальный лимит активных заказов на мастера.
•	adm:s:faq – отредактировать текст FAQ (возможно через загрузку файла или поэтапно).
•	adm:s:sup – изменить контакты поддержки (например, телефон/email).
•	adm:s:chan – настроить каналы/чаты (ввести новые ID для логов, алертов, отчетов).
•	Доступ и персонал (глобальный админ):
•	adm:staff:menu – меню управления персоналом.
•	adm:staff:list:<role>:<page> – список сотрудников по роли (GLOBAL_ADMIN, CITY_ADMIN, LOGIST).
•	adm:staff:new:<role> – создать нового сотрудника (генерирует код, затем предлагает выбрать города если роль городская: adm:staff:new:city:<city_id> повторяется для добавления нескольких городов, затем adm:staff:new:city:done для завершения).
•	После выбора городов: бот выдает код: adm:staff:code:<code_id> – показывает код и инструкцию передать сотруднику.
•	adm:staff:edit:<staff_id> – редактировать сотрудника (роль, ФИО, телефон – например, если ошибка).
•	adm:staff:edit:cities:<staff_id> – изменить привязанные города (для CITY_ADMIN/LOGIST).
•	adm:staff:deact:<staff_id> – деактивировать сотрудника (отзыв доступа, но оставить в списке).
•	adm:staff:revoke:<code_id> – отозвать выданный ранее код (если еще не использован).
(Пояснение: Все команды adm:* и m:* представляют callback_data кнопок. Пользователь также может вызывать основные разделы командами /start, /menu, но кнопочная навигация – основной способ взаимодействия.)

Приложение A — Формат логов распределения
Каждый цикл распределения (тик) логируется строкой, например:
[dist] order=1245 city=Москва district=ЮЗАО cat=ELECTRICS type=NORMAL  
round=1/2 sla=120s candidates=6  
ranked=[  
  {mid=78 shift=on car=1 avg_week=8200 rating=4.6 key=car>avg>rat>rnd(0.4)},  
  {mid=81 shift=on car=0 avg_week=9100 rating=4.9 key=car>avg>rat>rnd(0.1)},  
  ...  
]  
decision=offer mid=78 until=2025-09-15T10:02:00+03:00
Особые пометки могут добавляться в лог: - no_district – если у заявки не указан район (автораспределение не запускалось, заявка ушла логисту). - force_first – если кандидат принудительно поставлен первым (гарантийный случай, предыдущий мастер). - skip reason=off_shift|break|active_limit|... – если некоторые мастера были отфильтрованы и не попали в candidates, указывается причина.
Каждая строка лога отражает принятие решения на данном раунде. Если мастер отказался или не ответил, последующие тики будут логироваться аналогично с round=2/2 и др. Логи помогают анализировать работу алгоритма и при необходимости объяснять, почему тот или иной мастер не получил заказ.
Приложение B — Согласие на обработку персональных данных (тексты)
B.1. Для мастера (в боте):
Нажимая «Согласен», мастер подтверждает согласие на обработку и хранение персональных данных (ФИО, телефон, фото документа, селфи), а также информации о заказах и оплатах комиссий для целей модерации, допуска к заданиям, расчёта комиссий и безопасности сервиса.
Оператор сервиса — юридическое лицо, указанное в договоре-оферте сервиса. Сроки хранения — в период сотрудничества и в сроки, установленные законодательством. Передача третьим лицам — администраторам и логистам сервиса в необходимом объёме. Отзыв согласия возможен через поддержку; при отзыве доступ к заданиям прекращается.
Кнопки: Согласен / Не согласен
(Мастер должен нажать «Согласен» для продолжения регистрации. Нажатие «Не согласен» прекращает процесс.)
B.2. Для клиента (вне бота):
Согласие клиента на обработку его персональных данных (ПДн) получается вне системы (голосом по телефону, либо текстом при оформлении заявки через оператора). Примерный текст согласия, который оператор должен озвучить или отправить клиенту:
«Уважаемый клиент, оставляя заявку, Вы даёте согласие на обработку и хранение Ваших персональных данных: фамилии, имени, телефона, адреса и предоставленных материалов (фото, документы). Данные будут использованы исключительно для организации оказания услуг и обратной связи. Ваши данные могут быть переданы назначенному мастеру и администраторам сервиса в необходимом объеме. Срок хранения данных – согласно политике оператора сервиса и требованиям законодательства. Вы вправе отозвать согласие, обратившись в поддержку, однако в этом случае мы не сможем продолжить оказание услуг.»
Оператор фиксирует факт получения такого согласия (например, отметкой в системе). Без согласия клиента заявка не должна заводиться.
Приложение C — FSM сценарии (упрощенно, на базе Mermaid-диаграмм)
Для понимания реализованных последовательностей действий приводятся цепочки состояний для ключевых процессов:
•	Онбординг мастера:
CODE → PDN → FAM → NAM → MID → PHONE → CITY → DISTRICTS → CAR → SKILLS → PASS → SELFIE → PAYMETH → PAYDATA → HOMELOC → MODERATION
(Расшифровка: сначала ввод кода, затем согласие на ПДн, далее Фамилия, Имя, Отчество, Телефон, Город, выбор Районов, Наличие авто, выбор Навыков, Паспорт, Селфи, Способ выплат, Данные выплат, Домашняя локация (опц.), в конце – статус "на модерации".)
•	Создание заявки (админ):
START → CAT → ADDR → DESC → ATT → SLOT → CREATE → (SEARCHING | DEFERRED)
(Начало, выбор категории, адрес (город/район/улица/дом), описание, вложения, слот времени, создание заявки. После создания – либо сразу SEARCHING (если рабочее время), либо DEFERRED (если создано поздно вечером).)
•	Цикл выполнения заказа мастером:
OFFER → ASSIGNED → EN_ROUTE → WORKING → PAYMENT (акт+комиссия) → CLOSED
(Мастер получает оффер, принимает – заказ назначен. Далее отмечает «в пути», «работаю», закрывает заказ (ввод суммы и прикрепление акта переходят заказ в этап оплаты комиссии), после оплаты – закрыт.)
•	Комиссия (оплата мастером):
CREATED → WAIT_PAY → REPORTED → APPROVED | OVERDUE(блок)
(Создана при закрытии заказа; ожидает оплаты; мастер сообщил об оплате (reported); админ либо подтвердил (approved) – завершено, либо просрочено (overdue) – мастер заблокирован.)
(Эти диаграммы помогут при программировании FSM в библиотеке бота – нужно реализовать указанные переходы и ветвления.)
Приложение D — Развертывание и эксплуатация
Рекомендации по запуску системы и ее компонент:
•	База данных (PostgreSQL): разворачивается, например, через Docker. Команда для локального запуска: docker compose up -d postgres (предполагается, что docker-compose.yml настроен с сервисом postgres). Убедиться, что БД доступна по URL, указанному в переменных окружения.
•	Миграции и начальное заполнение: Запустить скрипт миграций (например, Alembic) для создания всех таблиц. Затем выполнить загрузку справочников географии: список городов, районов, улиц. Эти данные должны быть подготовлены (например, CSV-файлы) и загружены в соответствующие таблицы. В процессе импорта улиц использовать RapidFuzz для генерации ключей сравнения, как в исходной методике (чтобы поисковый функционал работал качественно). Также создать записи начальных админов/логистов (либо через .env ролей, либо вручную SQL).
•	Запуск ботов: Выполняются отдельными процессами (можно на одной машине):

 	python -m admin_bot.main  
python -m master_bot.main
 	Убедиться, что в логах появилось сообщение о старте и, при уровне debug, регистрация команд /start. Каждый бот должен подключиться к Telegram Bot API с соответствующим токеном.
•	Параллельный запуск: Не запускать несколько копий одного и того же бота. Если требуется перезапуск, старый процесс следует корректно завершить. При неправильном запуске двух экземпляров возможна ошибка 409 (конфликт getUpdates) и прекращение получения сообщений[2]. Лучше реализовать механизм блокировки (например, файл или запись в БД), предотвращающий двойной запуск.
•	Мониторинг: Убедиться, что настроены и корректно работают heartbeat сообщения (§18) – например, подписаться на LOGS_CHANNEL_ID и видеть периодические пинги. Также проверить отправку алертов: намеренно вызвать ошибку (например, принудительно бросить exception) – бот должен прислать сообщение с деталями.
•	Резервное копирование: Настроить cron-задачу или аналог для ежедневного вызова pg_dump. Проверить, что бэкап файлы создаются и доступны для восстановления. Желательно протестировать восстановление на тестовой среде.
•	Обновление и поддержка: При выкладке новых версий ботов нужно предусмотреть миграции БД (Alembic) и обратную совместимость данных. Боты должны быть остановлены на время миграции или использовать стратегии zero-downtime (сложно, поэтому обычно короткий простой).
После запуска провести полный цикл тестирования (см. чек-лист §20) уже на боевом окружении с тестовыми данными, чтобы убедиться, что все компоненты интегрированы правильно.
